Wikilife Meta API

REST Web Service Interface

The Wikilife Meta REST API allows you to query meta-data about life style variables.

Since the API is based on REST principles, it's very easy to write and test applications. You can use your browser to access URLs, and you can use pretty much any HTTP client in any programming language to interact with the API.

WikiGraph Concept

The wikilife Graph of meta data is a digraph or directed graph (more info about digraphs here). We use this data structure since it fix in the best way with the relationship between nodes.

  • Everything is a node that can be quantified.
  • Object-like oriented model using multi hierarchy and aggregation concepts.
Node

Each node has the information necessary to get a full description of it, using the relation with other nodes and its attributes.

Node attributes:

  • Mandatory attributes:
    • id, node identifier.
    • name, main name of a node.
    • metrics, list with at least one metric object.
  • Optional attributes:
    • otherNames, list of synonyms for that node.
    • is, list of node identifier.
    • has, list of has relation objects.
    • extras, extra information for that node.
Relation

There are two types relations between nodes in the wikiGraph:
"is" relationship and "has" relationship.

"HAS" Relation (aggregation):

Use a "has relation" object to define if a node is part of another node.
Example
>Apple Juice "has" Apple.

"IS" Relation (aggregation):

Use "is" to define if a node is a child of another node.
Example
Apple "is a" Fruit.

Metric

Nodes defines its own way to be quantified using Metrics Objects, it is possible to use more than one metric per node.

Base URL

All URLs referenced in the documentation have the following base:

http://meta.wikilife.org/

The Wikilife Meta REST API is open and free to be used for anyone.

About REST (REpresentational State Transfer)

We designed the Wikilfie Meta API in a very RESTful way, so that your consumption of it is simple and straightforward. From Wikipedia:

REST's proponents argue that the Web's scalability and growth are a direct result of a few key design principles:

  • Application state and functionality are divided into resources
  • Every resource is uniquely addressable using a universal syntax for use in hypermedia links
  • All resources share a uniform interface for the transfer of state between client and resource, consisting of
    • A constrained set of well-defined operations
    • A constrained set of content types, optionally supporting code on demand
  • A protocol which is:
    • Client-server
    • Stateless
    • Cacheable
    • Layered

REST's client/server separation of concerns simplifies component implementation, reduces the complexity of connector semantics, improves the effectiveness of performance tuning, and increases the scalability of pure server components. Layered system constraints allow intermediaries-proxies, gateways, and firewalls-to be introduced at various points in the communication without changing the interfaces between components, thus allowing them to assist in communication translation or improve performance via large-scale, shared caching.

REST enables intermediate processing by constraining messages to be self-descriptive: interaction is stateless between requests, standard methods and media types are used to indicate semantics and exchange information, and responses explicitly indicate cacheability.

If you're looking for more information about RESTful web services, the O'Reilly RESTful Web Services book is excellent.
